\# OrderHub - 注文書管理システム



\## プロジェクト概要

営業部門(東京・大阪・福岡)向けの注文書管理Webアプリケーション。

稼働しているエンジニアに対して有効な注文書が途切れなく存在する状態を維持し、

ギャップ(契約期間の途切れ)をリアルタイムに検知・管理するシステム。



\## 技術スタック(確定済み)

\- \*\*フロントエンド\*\*: Next.js (React, TypeScript) + Tailwind CSS

\- \*\*バックエンド\*\*: Next.js API Routes

\- \*\*認証\*\*: Firebase Authentication (Googleプロバイダ) → Google Workspace SSO

\- \*\*PDF解析\*\*: pdf.js のみ(OCR不使用・Cloud Vision API不採用)

\- \*\*データ/マスタ\*\*: Google Sheets API(稼働一覧・注文書台帳)

\- \*\*ファイル保存\*\*: Google Drive API(共有ドライブ)

\- \*\*ホスティング\*\*: Vercel

\- \*\*本番URL\*\*: https://orderhub-app-kappa.vercel.app/



\## 認証設計(確定済み)

\- 全社員がGoogle Workspaceアカウントを保有している前提

\- Firebase Auth の Googleプロバイダで SSO

\- 社内ドメインのみ許可(beat-tech.co.jp)

\- VPN不要、全拠点からHTTPS経由でアクセス

\- 追加費用なし(Firebase無料枠 50,000MAU、当社は数十名規模)



\## コスト(確定済み)

\- 月額ランニングコスト: 0円

\- 全APIがGoogle Workspace既存契約内または無料枠内

\### 2. アップロード画面(注文書登録・メタデータ付与)

\- PDF ドラッグ\&ドロップ → pdf.js でテキスト抽出 → フォーム自動入力

\- 対象区分切替: 社員番号 / BP番号 / multi(チーム)

\- 対象者選択で顧客コード・顧客名が自動補完

\- 手入力は契約期間(開始日・終了日)と発行日のみ

\- データなしPDF → 手入力フォームに切替え(OCR不使用)

\- ファイル名リアルタイムプレビュー



\## ギャップ検知ロジック(実データ検証済み)



\### データソース

\- \[A] 稼働一覧(Googleスプレッドシート): 当月稼働中のエンジニア一覧

\- \[B] 注文書台帳(本システム): 対象者ごとの契約開始日〜終了日



\### 突合ルール

1\. 稼働一覧の各拠点シートからD列(稼働)≧0.5の行を抽出 → \[A]

2\. 注文書台帳から contract\_start ≦ 当月 ≦ contract\_end を抽出 → \[B]

3\. \[A]にいるが\[B]にいない = ギャップ発生(赤アラート)

4\. E列(終了)=1.0 かつ 終了日が14日以内 = 期限迫る(黄アラート)



\### アラート分類

| ステータス | 条件 | 表示色 |

|-----------|------|--------|

| ギャップ発生 | 稼働中だが有効注文書なし | 赤 |

| 期限迫る | 契約終了14日以内 | 黄 |

| 正常 | 有効注文書の期間内 | 緑 |

| 当月終了 | E列=1.0 | オレンジ |

| アーカイブ候補 | 営業が申請済み | グレー |



\### 特殊ケース

\- 待機者: 完全にギャップ検知対象外(ダッシュボードに非表示)

\- 担当「全員」: 全営業のダッシュボードに表示

\- 稼働0.5(半稼働): 稼働中として検知対象

\- multi(チーム): 1つの注文書に複数名 → 全員を検知対象

\- 複数拠点同名(高山): メールアドレスでユニーク識別
