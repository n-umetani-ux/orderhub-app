## 1. 設計判断ログ（なぜそうしたか）

確定した設計判断と、その理由を時系列で記録する。理由を残すことで「なぜこの仕様にしたか」を後から追える。

| 決定事項 | 理由 | 決定日 |
|----------|------|--------|
| 顧客別フォルダの名前は **顧客コードのみ**（例: `C0105`）。顧客名は含めない | 顧客名を含めると表記ゆれ（全角/半角・カタカナ・略称・スペース有無）で同一顧客のフォルダが複数できてしまう。コードなら完全一致でゆれない | 2026-05-29 |
| 既存のフラット保存PDFは **顧客フォルダへ移行しない**（今後アップロード分のみ顧客フォルダへ） | 移行は重く・リスクがある。既存PDFの `driveLink` は親フォルダのURLのまま有効なので実害がない | 2026-05-29 |
| 同名注文書は **上書きせず2件目として共存** させ、保存前に確認ダイアログを出す | Google Drive は同名でも別ファイル（別ID）として扱う仕様。「上書き」は発生しない。差し替えなど正当なケースもあるため、ユーザーに確認させて続行可能にする | 2026-05-29 |
| 同名チェックの適用範囲は **multi・個人案件の両方**（全アップロード対象） | multi 起因の要望だったが、個人案件でも同名重複は起こりうるため全件チェックが安全 | 2026-05-29 |
| 重複チェックの検索範囲は **顧客フォルダ内 + 親フォルダ直下** の両方 | 既存フラット保存分（親フォルダ）と新規分（顧客フォルダ）の両方に同名がありうるため、両方を検索しないと取りこぼす | 2026-05-29 |
| 顧客フォルダの取得/作成・customerCode 取得に失敗した場合は **親フォルダにフォールバック** | 振り分けは「整理のための付加機能」であり、それが失敗してもPDF保存と台帳記録という本来の目的は必ず守る | 2026-05-29 |
| カバー〇のPDFリンクは **通常〇のみ**。オーバーライド（手動〇）は対象外 | オーバーライドは注文書が存在しないのに手動でカバー扱いにした状態で、紐づくPDFが存在しないため | 2026-05-29 |
| 401（トークン期限切れ）時のアップロードエラーは **再ログインを促す文言** を表示 | 「フォルダにアクセスできません」では原因が分からず問い合わせが増える。再ログインで自己解決できるよう案内する | 2026-05-29 |
| 突合キーは **manNo.（社員番号）** を基本とし、氏名は表示用と割り切る | 氏名は人間には同じに見えても別字のことがある（例: 「昴」と「昂」）。manNo. で突合すれば文字違いの影響を受けない | 2026-05-29 |
| 稼働一覧の読み取りは **サービスアカウント方式** | （既存決定。d5093fc / 2026-04-15） | 2026-04-15 |
| Drive書き込み・/api/sheets は **ユーザーOAuthトークン方式**（SA化は将来課題） | 既存実装を踏襲。ただし約1時間でトークンが切れる制約があり、Phase 6 前に根本対処を検討 | （既存） |
| 同名・同一契約期間の注文書は **error ではなく warn** とし、ユーザー確認で2件目を登録可能にする | 差し替え・意図的な2件目登録など正当なケースがある。error で完全ブロックすると、登録後に手作業で統合・オーバーライドする運用が回らない。登録を止める error は「契約開始日≧終了日」の論理エラーのみに限定（コミット a5b748c） | 2026-06-16 |
| 同一manNo・**同一顧客**で契約期間が重なる注文書は **uploadedAt が最新の行を正** とし、それに隠れる古い行をギャップ判定から除外（dedup）。グループキーは **manNo+customerCode** | 契約短縮・差し替えで2件目を登録したとき、古い広い注文書が台帳に残るとカバー判定（OR）が古い行に引っ張られ、本来ギャップの月を緑表示して**見逃す**。最新を正とすれば短縮・延長とも正しく反映。別顧客の並行稼働（半稼働0.5×2社など）は別系列として保持するため customerCode をキーに含める。uploadedAt 欠損時は台帳の行順（後方＝新しい）で代替（コミット 864c80a） | 2026-06-16 |
| `MONTHLY_SHEET_IDS` を設定シート（既存 key/value）経由でも読めるようにし、管理者がUIから月別シートIDを登録可能にする（中間案・方式B） | 毎月のシートID追記＝コミット＆デプロイの運用負荷を解消。getActiveSheetIds は純関数のまま overrides を引数で受け取りI/O化しない（読み取り失敗で全停止しないため）。フォールバック順序は 設定シート → ハードコード → env（当月のみ）。最悪時も現行ハードコード動作に縮退（コミット 169c8a7・bf82aa4） | 2026-06-16 |
| 月別シートID登録は保存前に検証エンドポイント（/api/sheets/validate）で「稼働表タブを持つ正しいシートか」を確認してから保存する | 誤ったID（タイポ・別シート）を登録するとその月が無言でギャップ化する事故を防ぐ。検証のタブ判定は実読み取りと同じ SHEET_LOCS 完全一致基準（src/lib/sheet-locations.ts に単一正本化）。月選択は年月ドロップダウンで形式ミスを構造的に防止。検証は POST + body 化し sheetId をログに残さない | 2026-06-16 |
| 管理者向け仕様書ページを追加（会社説明用） | 非エンジニアの管理者が技術・費用・セキュリティの説明要求に対応できるよう、仕様書をアプリ内で参照可能に。内容は構造化データ（spec-content.ts）で区切りごとに更新。新規ライブラリ非依存（firebase-admin の本番ロード失敗の教訓）。isAdmin 二重ゲートで管理者限定（コミット 0ecc438） | 2026-06-16 |

## 2. 運用ルール（人が守る・マニュアルで対応すること）

システムではなく、運用・マニュアルで担保するルール。

| ルール | 内容 | 根拠/経緯 |
|--------|------|-----------|
| 氏名の表記は **台帳（稼働一覧）を正** とする | 注文書PDFと台帳で氏名表記が違う場合、台帳を正とし、注文書PDF側を運用マニュアルで修正する | 李氏の「昴蔚（PDF）」vs「昂蔚（台帳）」の文字違い対応（2026-05-29） |
| トークン期限切れ時は **再ログイン** で対処 | アップロード時に「ログインの有効期限が切れました…」と出たら、一度ログアウト→再ログインしてから操作 | OAuthアクセストークンが約1時間で失効するため |
| 機密情報は **値を伏せて** 扱う | `.env.*` の中身・サービスアカウントJSON・private_key・環境変数のValue・APIシークレットはチャット等に貼らない。確認は「XX変数はある?(yes/no)」形式で | 2026-05-18 の秘密鍵漏洩ヒヤリハットの教訓 |
| アーカイブは **営業が申請 → 月末に管理者が一括確認** | 営業がダッシュボードから申請し、月末に管理者がまとめて確認・確定する | （既存の権限設計） |
| 編集・アップロードは **自分の担当案件のみ** | 閲覧は全拠点可能だが、編集・アップロードは担当者本人のみ | （既存の権限設計） |
| 契約期間の**短縮・差し替え**時は古い台帳行を**手動削除**する | 2件目を登録後、古い行を削除する。ダッシュボードのギャップ判定からは自動除外（dedup）されるが、台帳の生データには古い行が残る（削除対象を画面で見えるよう、表示はあえて dedup しない設計）。延長時は古い行を残しても表示は正しいが、短縮時は削除しないと見た目の台帳が冗長になる | dedup 導入（2026-06-16）に伴う運用 |
| 月初に**新しい月の稼働一覧シートIDを `MONTHLY_SHEET_IDS` に登録**する | 新しい月の稼働一覧スプレッドシート作成後、そのシートIDをソースコード（src/lib/monthly-sheets.ts）に追記しコミット＆デプロイ。getActiveSheetIds() は当月+翌月まで走査するため、登録すれば即取り込み対象になる | MONTHLY_SHEET_IDS がハードコードのため（将来UI化で解消予定。6章参照） |
| 月初の新月シートID登録は **管理者設定画面から**行う（コミット・デプロイ不要） | 管理者設定モーダルの「月別 稼働一覧シートID」で年月を選びシートID/URLを入力 →「検証して追加」。稼働表タブの存在を検証後に登録・即反映。既存月（4〜7月）はハードコードのまま、8月以降を画面登録する運用 | MONTHLY_SHEET_IDS UI化（2026-06-16・Phase 6前の運用負荷削減） |
| 仕様書（会社説明用）は **区切りごとに** spec-content.ts を更新する | 毎回の細かい変更は載せず、節目で更新。最終更新日は SPEC_LAST_UPDATED で管理 | 仕様書ページ追加（2026-06-16） |

## 3. データの扱い・突合ルール（確定仕様）

### 機密情報の隔離
稼働一覧シートの **I列（売上）・O列（原価）・X列（粗利）および J〜W列** は API 層で完全除外。フロントエンドには絶対に渡さない。

### ギャップ検知の突合
- 突合キーは **manNo.（6桁数値）**
- 稼働一覧の各拠点シートから D列（稼働）≧0.5 を抽出
- 注文書台帳から contract_start ≦ 当月 ≦ contract_end を抽出
- 稼働中だが有効注文書なし = ギャップ（赤）／契約終了14日以内 = 期限迫る（黄）

### 複数注文書の世代解決（dedup）
同一manNo・同一顧客で契約期間が重なる注文書が台帳に複数あるとき、ギャップ/カバレッジ計算では **uploadedAt 最新の1件を正** とし、隠れる古い行を除外する（`selectEffectiveOrders`・gap-detector.ts）。
- グループキー: **manNo + customerCode**（別顧客の並行稼働は別系列で保持）
- 世代判定: uploadedAt 降順 → 欠損/同値時は台帳行順（後方＝新しい）
- 適用範囲: **カバレッジ/ギャップ計算のみ**。台帳行の表示・Driveリンクは生のまま（手動削除のため古い行を残す）
- 重ならない連続契約（4-6月＋7-9月など）は両方保持
- 既知の許容リスク: ①過去の注文書を後から登録（バックフィル）すると uploadedAt が新しくなり世代誤認 ②契約日付が不正文字列だと dedup が no-op。いずれも結果は「誤ギャップ（赤＝アラート方向）/ フェイルオープン」で安全側のため許容

### 注文書台帳の主なカラム
manNo / name / contractStart / contractEnd / fileName / **driveLink（webViewLink）** / dept / customerCode / customerName / targetType / uploadedAt / uploadedBy

## 4. ファイル/フォルダ構成ルール（確定仕様）

### 命名規約
- 個人参画: `YYMMDD_部署コード_顧客コード顧客略称_社員番号orBP番号.pdf`
- チーム参画: `YYMMDD_部署コード_顧客コード顧客略称_multi-連番.pdf`
  - 連番はユーザーがドロップダウンで手動選択（0=1枚目、1以降=2枚目以降）

### Drive保存先（2026-05-29〜）
- 親フォルダ直下に **顧客コード名のサブフォルダ**（例: `C0105`）を自動作成し、そこへ保存
- 顧客フォルダが無ければ作成、あれば再利用（作成後に再検索して最初のIDを採用＝並行作成対策）
- customerCode が空 / API失敗時は親フォルダ直下に保存（フォールバック）
- 既存のフラット保存分は移行しない

## 5. 既知の制約・ハマりポイント

| 項目 | 内容 | 対処 |
|------|------|------|
| OAuthトークンの寿命 | `/api/drive`・`/api/sheets` はユーザーOAuthトークンで認証し、約1時間で失効 | 再ログイン（恒久対処はPhase 6前に検討） |
| `.next` キャッシュ汚染 | ローカルで `GET / 404` 等の不可解なエラーが出ることがある | `Remove-Item -Recurse -Force .next` → `npm run dev`（Windows/PowerShell） |
| Drive フォルダID | env変数の値に末尾改行が混入しやすい | 読み込み時に `trim()` する |
| 共有ドライブの検索/作成 | `files.list`/`files.create` で共有ドライブを対象にするには `supportsAllDrives: true` と `includeItemsFromAllDrives: true` が必要 | 両方を指定する |
| サーバー側ログの確認場所 | `console.log` はブラウザF12には出ない | 本番=Vercel Logs（Runtime）/ ローカル=PowerShellのターミナル |
| L1/L2キャッシュ（72hTTL）の反映遅延 | シートID登録やデータ更新が即座にダッシュボードへ出ない場合がある | デプロイ後、管理者がダッシュボードで再同期するとキャッシュ更新される |
