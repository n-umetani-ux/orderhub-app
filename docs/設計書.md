# 設計書: 注文書管理システム（OrderHub）

## 1. システム概要

### 1.1 目的
営業部門（東京・大阪・福岡）が管理するエンジニアの注文書（契約書）について、契約期間の途切れ（ギャップ）をリアルタイムに検知・管理するWebシステム。

### 1.2 対象ユーザー
- 営業担当者（東京ITS営業部、東京ENG営業部、大阪営業部、福岡営業部）
- 管理者（システム設定の変更権限あり）

### 1.3 利用環境
- ブラウザ: Chrome/Edge推奨
- VPN不要、HTTPS経由
- Google Workspaceアカウント必須（@beat-tech.co.jp）

## 5. データ設計

### 5.1 稼働一覧（月別スプレッドシート）

月ごとに別ファイル。常に当月＋翌月の2ファイルを同時読み込み。

参照シート: 稼働表（東京）、稼働表（大阪）、稼働表（福岡）

| 列 | 項目 | 用途 |
|----|------|------|
| A | manNo. | 社員番号（6桁、突合キー） |
| B | 区分 | エンジニア種別 |
| C | 技術者名 | 表示用氏名 |
| D | 稼働 | 稼働値（1.0/0.5/0.0） |
| E | 終了 | 当月終了フラグ（1.0/空） |
| F | 顧客名 | 取引先名 |
| G | 担当 | 営業担当者名 |
| AE | 顧客コード | C+4桁（命名規約用） |

除外対象:
- 待機一覧セクションの技術者
- D列（稼働）が0.5未満の行
- I列〜X列（売上・原価・粗利等の機密情報）→ API層で完全除外

月別スプレッドシートID管理: src/lib/monthly-sheets.ts

新しい月のシートが作成されたら、ここに1行追加する。

### 5.2 注文書台帳シート

## 6. API設計

### 6.1 GET /api/sheets
稼働一覧からエンジニア情報を取得

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token |
| レスポンス | engineers: Engineer[], loadedMonths: string[] |
| 処理 | 当月＋翌月のシートを並列読み込み → パース → 重複除去（翌月優先） → ギャップ検知 → activeMonths付与 |

### 6.2 GET /api/orders
注文書台帳の全レコードを取得

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token |
| レスポンス | orders: OrderRecord[] |

### 6.3 POST /api/orders
注文書台帳に新規レコードを追加

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token |
| リクエスト | manNo, name, contractStart, contractEnd, fileName, driveLink, dept, customerCode, customerName, targetType, uploadedBy |
| レスポンス | ok: true |

### 6.4 POST /api/drive
PDFをGoogle Driveにアップロード

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token |
| リクエスト | FormData（file, fileName） |
| レスポンス | link: string |
| 備考 | 保存先フォルダは設定シートから取得（未設定なら環境変数） |

### 6.5 GET/POST /api/overrides
手動オーバーライドの取得・設定

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token |
| POST | manNo, yearMonth, status (covered/gap/na/auto), updatedBy |
| 備考 | status="auto"でオーバーライド解除 |

### 6.6 GET/POST /api/settings
管理者設定の取得・変更

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token + x-user-email |
| GET | settings: {}, isAdmin: boolean |
| POST | key, value（管理者のみ、403で拒否） |

### 6.7 POST /api/archive
アーカイブ申請

| 項目 | 内容 |
|------|------|
| 認証 | x-google-access-token |
| リクエスト | manNo, name, requestedBy |

## 7. ギャップ検知ロジック

### 7.1 ステータス判定（自動）

1. アーカイブ申請済み → "archived"（グレー）
2. 終了フラグ(E列)=1.0 → "ending"（オレンジ）
3. 有効な注文書なし（契約期間が今日を含まない）→ "gap"（赤）
4. 契約終了日まで14日以内 → "expiring"（黄）
5. 上記いずれでもない → "normal"（緑）

### 7.2 月別カバレッジ判定（ダッシュボード表示）

各エンジニア × 各月に対して:

1. 手動オーバーライドあり → オーバーライド値を優先
2. その月をカバーする注文書あり → "covered"（○）
3. その月の稼働一覧に存在 → "gap"（未）
4. その月の稼働一覧に非存在 → "na"（—）

### 7.3 サマリー集計

| カード | 計算方法 |
|--------|---------|
| 当月ギャップ | 当月稼働対象で当月注文書なしの人数 |
| 来月切れ | 当月カバー済みだが翌月注文書なしの人数 |
| 正常カバー | 当月・翌月ともカバー済みの人数 |
| 監視中 | アーカイブ除く全稼働者数 |

## 8. PDF解析・命名規約

### 8.1 PDF解析
- pdfjs-dist でブラウザ内テキスト抽出（OCR不使用）
- 日付パターンマッチ: 「2026年4月1日〜2026年6月30日」「2026/04/01-2026/06/30」
- テキスト抽出不可の場合 → 手入力フォームに切替

### 8.2 ファイル命名規約

個人参画:
YYMMDD_部署コード_顧客コード顧客略称_社員番号.pdf
例: 260401_1010_C0105SCSK_170156.pdf

チーム参画（multi）:
YYMMDD_部署コード_顧客コード顧客略称_multi.pdf （1枚目）
YYMMDD_部署コード_顧客コード顧客略称_multi-1.pdf （2枚目以降）
